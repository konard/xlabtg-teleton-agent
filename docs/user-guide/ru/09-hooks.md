# Хуки (Hooks)

Hooks — это пред-ответные преобразования: они читают каждое входящее сообщение и либо *блокируют* его, либо *подмешивают контекст* перед ответом агента, либо *запускают* действие. Это самый лёгкий способ навязать ключевые правила, не меняя промптов и не дописывая код.

![Страница Events для проверки hook-событий](../assets/screenshots/ru/events-page.png)

## Структура страницы

В шапке — переключатель режима:

| Режим | Когда применять |
| --- | --- |
| **Basic** | Однострочные блок-листы и контекстные триггеры. Быстрее всего настроить. |
| **Advanced** | Полноценный визуальный конструктор с condition-блоками и action-блоками. |

Под переключателем — **Test panel**: «песочница», в которую можно вставить сообщение и увидеть, какие хуки сработают. Сохранение происходит с дебаунсом ~400 мс — править можно без ручных кликов «save».

## Basic-режим

В Basic есть две простые сущности.

### Keyword blocklist

Блок-лист отклоняет сообщения, содержащие любое из заданных слов, и возвращает фиксированный ответ. Применение — жёсткие стопы:

- seed-фразы, приватные ключи, мнемоники;
- запрещённые темы поддержки («когда mooon», «гарантирую x10»);
- слова, которые не должны попадать в LLM (паттерны PII, внутренние кодовые имена).

Для каждого ключевого слова задаётся:

- **Match** — substring (по умолчанию), word boundary или regex.
- **Response** — текст, который дословно отправляется пользователю.
- **Audit reason** — метка, которая попадает в [Центр безопасности](08-security.md) при срабатывании.

> ℹ️ **Будьте конкретны.** «key» — слишком широко, перекроет легитимные вопросы. «private key» или regex `\\bseed\\b` куда безопаснее.

### Context triggers

Контекстные триггеры — обратное блокировке: они **добавляют инструкции** агенту перед ответом. Пример: триггер на `airdrop` подмешивает напоминание *«Предупреди о скам-рисках; никогда не советуй подключать кошелёк к незнакомым сайтам.»*.

Для каждого триггера:

- **Keyword / regex** — те же опции совпадения, что и у блок-листа.
- **Injected context** — markdown, добавляемый в системный промпт для этого ответа.
- **Scope** — только DM, только группы или оба варианта.

## Advanced-режим (визуальный конструктор)

Advanced собирает упорядоченные правила из **condition-блоков** и **action-блоков**:

| Condition | Срабатывает, когда… |
| --- | --- |
| `keyword` | Сообщение содержит подстроку или regex. |
| `chat_type` | Чат `dm`, `group` или `channel`. |
| `chat_id` | ID чата входит в список. |
| `user_id` | Отправитель входит в список. |
| `time_window` | Текущее время в заданном окне. |

| Action | Эффект |
| --- | --- |
| `block` | Отказ с фиксированным ответом. |
| `inject_context` | Добавить markdown в системный промпт текущего хода. |
| `route_to_tool` | Пропустить LLM и вызвать инструмент напрямую. |
| `notify` | Бросить событие в [Events](10-advanced-features.md#events). |
| `escalate` | Запросить человеческое подтверждение через [Центр безопасности](08-security.md). |

Правила применяются сверху вниз; побеждает первое совпавшее, за исключением `notify` — он не блокирующий и складывается со следующим правилом.

## Test panel

![Запись в audit trail об изменении хука](../assets/screenshots/ru/audit-trail-security-page.png)

Перед сохранением нового набора хуков всегда проверяйте его:

1. Вставьте репрезентативное сообщение.
2. Выберите тип чата (`dm`, `group`, `channel`) и, опционально, ID отправителя.
3. Нажмите **Run**.
4. Прочитайте отчёт:
   - какие хуки сработали (в порядке);
   - заблокировано ли сообщение;
   - какой контекст подмешан;
   - финальное действие.
5. Поправьте ключевые слова и правила, чтобы убрать ложные срабатывания, и прогоните снова.

Запустите хотя бы один **позитивный** случай (сообщение, которое мы хотим ловить) и хотя бы один **негативный** (сообщение, которое должно пройти без вмешательства).

## Где хуки уместны (и где нет)

Хуки — это **локальный контроль поведения** для входящих сообщений. Они не подходят для:

- **Исходящей автоматизации** — для этого используйте [Workflows](10-advanced-features.md#workflows) (по cron/событию) и [Webhooks](10-advanced-features.md#events).
- **Внешних интеграций** — открывайте страницу [Integrations](10-advanced-features.md#integrations).
- **Жёстких политик** — используйте [Security Center → Policies](08-security.md). Хук может подсказать поведение; политика — заставит.

## Правила проектирования

- **Будьте конкретны.** «private key» лучше, чем «key». Regex с границами слов лучше «голых» подстрок.
- **Блокируйте секреты, советуйте остальное.** `block` — для seed/ключей/абьюза. `inject_context` — для рекомендаций и дисклеймеров.
- **Тестируйте позитив и негатив.** Каждому новому хуку нужен хотя бы один пример того и другого.
- **Не плодите слои.** Несколько правил в правильном порядке проще понять, чем десятки перекрывающихся.
- **Аудит после правок.** Откройте [Security Center → Audit Trail](08-security.md) и проверьте, что событие `hook_change` записалось.
