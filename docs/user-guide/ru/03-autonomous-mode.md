# Автономный режим (Autonomous Mode)

Autonomous Mode — страница, на которой Teleton Agent выполняет длительные цели самостоятельно. Агент идёт по циклу *plan → act → observe → reflect → checkpoint*, сохраняет состояние на диск (поэтому продолжает работу после перезапуска) и показывает каждый шаг оператору. По умолчанию режим **выключен**; включайте его только после того, как настроены политики в [Настройках (Configuration)](11-settings.md) и заполнены admin IDs.

![Список автономных задач](../assets/screenshots/ru/autonomous-page-initial.png)

## Структура страницы

| Зона | Что в ней |
| --- | --- |
| **Шапка** | Глобальный переключатель автономного выполнения; индикатор лимита параллельных задач. |
| **+ New task** | Открывает форму создания задачи с естественно-языковой строкой и структурированной формой. |
| **Таблица задач** | Колонки: ID, Status, Goal, прогресс-бар, Priority, Strategy, Created, Last update, кнопки действий (Start / Pause / Cancel / Delete). |
| **Панель деталей** | Открывается по щелчку на строке; вкладки *Goal & constraints*, *Live log*, *Checkpoints*, *Result*, *Errors*. |
| **Фильтры** | Статус (any / pending / running / paused / completed / failed / cancelled), стратегия, поиск по ID или подстроке цели. |

## Жизненный цикл задачи

![Конечный автомат автономной задачи](../assets/diagrams/autonomous-state-machine.svg)

Состояния и переходы:

| Состояние | Что значит | Возможные переходы |
| --- | --- | --- |
| `pending` | Задача создана, но ещё не подхвачена планировщиком. | → `queued`, → `cancelled` |
| `queued` | Ждёт свободный слот исполнителя. | → `running`, → `cancelled` |
| `running` | Активный цикл plan/act/observe. | → `paused`, → `completed`, → `failed`, → `cancelled` |
| `paused` | Приостановлена оператором, чекпоинт сохранён. | → `running`, → `cancelled` |
| `completed` | Цель достигнута, success criteria выполнены. | конечное |
| `failed` | Сработало условие провала или превышен бюджет. | конечное |
| `cancelled` | Задача остановлена оператором. | конечное |

Запущенная задача (`running`) пишет событие `checkpoint` примерно на каждой итерации. После перезапуска агента задачи в состояниях `queued`/`running` автоматически продолжаются с последнего чекпоинта без вмешательства оператора.

## Создание задачи через AI-парсер

![Форма создания задачи](../assets/screenshots/ru/autonomous-create-form.png)
![Парсер естественного языка — успех](../assets/screenshots/ru/nl-parser-success.png)

1. Нажмите **+ New task**.
2. В верхнем поле опишите цель простыми словами. Укажите триггер («каждые 5 минут», «когда происходит X»), действие, адресата (чат, канал, файл) и критерии успеха/провала.
3. Нажмите **Parse with AI**. Правая колонка заполнится структурированными полями; в правом верхнем углу появится метка **confidence**.
4. Сравните содержимое полей с исходным описанием:
   - **Goal** — одно предложение в действительном залоге.
   - **Success criteria** — по одному пункту в строке; каждый должен быть измеримым.
   - **Failure conditions** — когда цикл должен остановиться, а не повторять попытку.
   - **Allowed tools / Restricted tools** — явные белые и чёрные списки инструментов.
   - **Strategy** — `conservative`, `balanced` или `aggressive`. Conservative делает меньше итераций и отказывается от неоднозначных вызовов инструментов.
   - **Priority** — от 1 (низкий) до 10 (высокий). Более высокие приоритеты вытесняют низкие в очереди.
   - **Iteration limit** — жёсткий лимит итераций цикла планирования.
   - **Duration limit** — лимит реального времени работы.
   - **Budget** — предел расхода токенов и предел расхода TON.
5. Поправьте всё, что не соответствует вашему намерению. **Всегда проверяйте restricted tools** перед сохранением.
6. Нажмите **💾 Save & Start** (или **Save**, чтобы оставить задачу в `pending`).

Если уверенность ниже 70%, парсер показывает предупреждающий баннер. Перепишите описание или заполните поля вручную; не сохраняйте задачу с низкой уверенностью.

![Поток создания задачи](../assets/diagrams/task-creation-flow.svg)

## Создание задачи вручную

Откройте **+ New task** и работайте только со структурированной формой, когда нужны точные ограничения:

- По одному пункту success criterion на строку.
- По одному пункту failure condition на строку.
- Высокорисковые инструменты (`ton_send`, `jetton_send`, `exec_run`, `workspace_delete`) помещайте в **restricted tools**, если цель явно их не требует.
- Для работы с кошельком или аккаунтом выбирайте `conservative` и небольшой TON-бюджет.

## Наблюдение за выполнением

![Панель деталей задачи](../assets/screenshots/ru/autonomous-detail-panel.png)

Кликните на строку в таблице — откроется панель деталей. Вкладка **Live log** показывает структурированные события по мере появления:

| Событие | Что значит |
| --- | --- |
| `plan` | Агент выбрал следующее действие и обосновал его. |
| `tool_call` | Вызван инструмент с указанными аргументами. |
| `tool_result` | Инструмент вернул значение, ошибку или отказ. |
| `reflect` | Агент оценил прогресс относительно success criteria. |
| `checkpoint` | Состояние сохранено на диск. |
| `escalate` | Требуется человеческое подтверждение (см. [Центр безопасности](08-security.md)). |
| `info` | Свободный комментарий: подсказка парсера, заметка планировщика. |
| `error` | Шаг завершился ошибкой. Дальнейшее поведение зависит от стратегии. |

У каждого события есть метка времени, индекс итерации и стрелка раскрытия с полным телом полезной нагрузки. Длинные аргументы и результаты сворачиваются — щёлкните по строке, чтобы увидеть полное тело.

## Pause, Resume, Cancel

| Действие | Когда применять |
| --- | --- |
| **Pause** | Задаче нужен дополнительный контекст, изменение политики или дополнительные согласования. Возобновится с последнего чекпоинта. |
| **Resume** | После Pause, когда правка внесена. Следующая итерация стартует с сохранённого плана. |
| **Cancel** | Цель больше не нужна или задача ушла «не туда». Cancelled — конечное состояние; пересоздавайте задачу заново вместо оживления. |
| **Delete** | Убирает задачу и её лог из таблицы. События аудита остаются в [Центре безопасности](08-security.md). |

## Правила безопасности

- Поддерживайте заполненным `telegram.admin_ids`. Пустой список admin блокирует эскалации и не даёт включиться автономному режиму.
- Для любых задач с кошельком, аккаунтом или `exec` предпочитайте стратегию **`conservative`**.
- Для задач, которые могут переводить средства, явно задавайте **TON-бюджет**. Агент остановится на границе бюджета, а не попросит увеличить его.
- Ограничивайте инструменты, которые переводят средства, пишут файлы или ходят во внешние сервисы, если цель явно не требует их использования.
- Перед возобновлением эскалированной задачи откройте **Security Center → Approvals**.
- После завершения задачи изучите вкладку **Result** — там итоговые рассуждения, собранные доказательства и оставшиеся открытые пункты.

## Рекомендуемые паттерны

- **Периодический мониторинг** — короткий success criterion («сообщить, когда количество > N»), стратегия conservative, низкий приоритет, длительность 24 часа.
- **Одноразовое исследование** — стратегия balanced, небольшой iteration limit, без Telegram-инструмента отправки; отчёт пишется в workspace, а уведомлять чат поручают отдельной задаче.
- **Операции с кошельком** — стратегия conservative, разрешён ровно один кошельковый инструмент, маленький бюджет, эскалация при первой ошибке.
