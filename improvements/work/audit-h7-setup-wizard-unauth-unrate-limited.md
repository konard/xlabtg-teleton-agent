## Источник

Найдено в ходе аудита кодовой базы — issue #250, отчёт [`AUDIT_REPORT.md`](../../AUDIT_REPORT.md) (AUDIT-H7).

## Описание

`POST /api/setup/launch` на порту 7777 генерирует случайный токен, записывает его **в plaintext** в `config.yaml` и возвращает в теле ответа. Сервер setup'а привязан к `127.0.0.1:7777`, но **без аутентификации, CSRF и rate-limit**.

Всё, что работает на том же хосте — расширения браузера, другие процессы, локальная малварь, docker-контейнер, шарящий network namespace — может:
- дёргать этот эндпоинт циклично, **ротируя токен** и локаут настоящего пользователя;
- **собирать** новый токен из ответа.

## Местоположение

`src/webui/setup-server.ts:132-161`

## Влияние

Конфиденциальность и доступность setup-флоу. При компрометации хоста локальный злоумышленник может захватить setup-процесс без необходимости повышать привилегии.

## Предложенное исправление

1. Требовать у `POST /api/setup/launch` **одноразовый bootstrap-nonce**, который CLI печатает при старте (`--setup-nonce ...` или в stderr как одноразовая ссылка).
2. Добавить простой in-process rate limiter (например, 5 запросов в минуту на клиента).
3. Хранить токен в config **солёным хэшем** (bcrypt / scrypt) и сравнивать хэши — так утечка config не равна утечке токена.
4. Привязать сессию к конкретному процессу-клиенту (например, через unix socket вместо TCP, если возможно).

## Критерии приёмки

- [ ] `POST /api/setup/launch` отклоняет запрос без валидного одноразового nonce.
- [ ] Nonce печатается CLI в stderr и не логируется в stdout.
- [ ] Rate-limiter: не более 5 запросов/минуту с одного IP (или отказ после первой успешной инициализации).
- [ ] Токен в `config.yaml` хранится как хэш; при проверке сравнение с хэшем.
- [ ] Юнит-тест: запрос без nonce → 401/403.
- [ ] Юнит-тест: после `N + 1` запросов rate-limiter блокирует следующий.
- [ ] Юнит-тест: токен не восстанавливается из config.yaml напрямую.

## Оценка

**Effort:** small (≈ 4–6 часов с тестами).
**Priority:** P1 — fix before re-enabling autonomous wallet mode.
